# Local Large Artifacts

## Purpose

This document lists the large local-only files created or consumed by the PubChem bulk Tier D workflow, executor workflows, and local database indexing. These files are needed for reproducible bulk screening or fast local querying on a workstation, but they should not be pushed to the normal GitHub repository history.

The main reasons are:

- the raw PubChem FTP archives are very large
- the bulk screening parquet outputs are regenerated artifacts
- the local DuckDB index can exceed GitHub's single-file size limit after rebuilds
- these files would make normal Git pushes impractical and can exceed GitHub-friendly repository size

## Files That Should Not Be Committed

### Raw PubChem FTP inputs

These are official PubChem `Compound/Extras` downloads stored under `data/raw/pubchem_bulk/extras/`.

| Path | Current local size | What it is | Why keep local-only |
| --- | ---: | --- | --- |
| `data/raw/pubchem_bulk/extras/CID-SMILES.gz` | `1.38 GiB` | `CID -> SMILES` mapping used for RDKit screening | raw upstream archive, reproducible from PubChem FTP |
| `data/raw/pubchem_bulk/extras/CID-InChI-Key.gz` | `6.81 GiB` | `CID -> InChI / InChIKey` mapping used for identity reconciliation | raw upstream archive, largest single bulk input |
| `data/raw/pubchem_bulk/extras/CID-Mass.gz` | `1.29 GiB` | `CID -> formula / monoisotopic mass / exact mass` used for DuckDB coarse prefilter | raw upstream archive, cheap-to-regenerate download |
| `data/raw/pubchem_bulk/extras/CID-Component.gz` | `0.12 GiB` | component membership used to exclude multi-component entries | raw upstream archive |
| `data/raw/pubchem_bulk/extras/CID-Synonym-filtered.gz` | `0.90 GiB` | filtered synonym list used to attach primary names | raw upstream archive |

These files are downloaded by:

```powershell
.venv\Scripts\python.exe pipelines\acquire_pubchem_bulk_candidates.py --download-only
```

### Generated bulk-screening outputs

These files are derived from the raw FTP inputs and can be rebuilt locally.

| Path | Current local size | What it is | Why keep local-only |
| --- | ---: | --- | --- |
| `data/bronze/pubchem_candidate_pool.parquet` | `60.97 MiB` | deduplicated surviving candidate pool after DuckDB prefilter + RDKit screening | generated artifact, rebuildable |
| `data/bronze/pubchem_candidate_filter_audit.parquet` | `12.88 MiB` | audit table for the materialized bulk-screening universe | generated artifact, rebuildable |

These files are generated by:

```powershell
.venv\Scripts\python.exe pipelines\acquire_pubchem_bulk_candidates.py --skip-download
```

### Local PubChem request cache

Per-request PubChem JSON cache files created during Tier D lookup retries are local-only:

- `data/raw/pubchem/tierd_excel202603_*.json`

They are not release source-of-truth. Use `data/raw/generated/pubchem_tierd_candidates.csv` and the rebuilt seed catalog for reviewable Tier D state.

### Local quantum and cycle executor artifact trees

Executor working directories can be much larger than the reviewable release
artifacts. Keep the directories below local-only and commit only the small CSV
manifests/results plus rebuilt Parquet/DuckDB outputs that reference their
checksums.

| Path | Current local size | What it is | Why keep local-only |
| --- | ---: | --- | --- |
| `data/raw/manual/quantum_phase2_artifacts/` | `~16 GiB` | xTB Hessian and CREST phase-2 executor bundles | large local executor scratch/output tree; checksums are already recorded in committed CSV/Parquet outputs |
| `data/raw/manual/quantum_pilot_artifacts/` | `~294 MiB` | xTB/Psi4 quantum pilot executor bundles | rebuild/verification evidence is represented by `quantum_pilot_results.csv`, `quantum_job`, and `quantum_artifact` |
| `data/raw/manual/cycle_backend_artifacts/` | small/local | optional CoolProp cycle backend scratch artifacts | local execution support files, not a release contract |

Generated XYZ input directories under `data/raw/generated/` are also local-only
because they can be regenerated from their CSV request/XYZ manifests:

- `data/raw/generated/quantum_xyz/`
- `data/raw/generated/quantum_dft_xyz/`
- `data/raw/generated/governance_xtb_xyz/`
- `data/raw/generated/governance_dft_singlepoint_xyz/`
- `data/raw/generated/governance_phase2_xyz/`

### Local DuckDB index snapshots

The DuckDB files under `data/index/` are convenience query indexes over the
reviewable CSV/Parquet layers. They are not source-of-truth: they can be rebuilt
from `data/raw`, `data/bronze`, `data/silver`, and `data/gold`.

| Path | Current local size | What it contains | GitHub-visible substitute |
| --- | ---: | --- | --- |
| `data/index/r_physgen_v2.duckdb` | `~106 MiB` after the v1.6.3 residual rebuild | DuckDB copy/index of the current layered dataset, including source manifest, molecule/property tables, quantum audit tables, active-learning queue, and gold model tables | Keep regenerated snapshots local when they exceed GitHub-friendly limits; use committed Parquet/CSV tables plus `quality_report.json`/`validation_report.json` to inspect the same dataset scope |
| `data/index/r_physgen_v2.v1.6.3-draft.local.duckdb` | local-only optional backup | Local backup of the rebuilt v1.6.3 draft DuckDB index before upload-history cleanup | Documented here; not required for GitHub review |

If a GitHub push would include a newly rebuilt `data/index/r_physgen_v2.duckdb`
larger than the platform limit, exclude that binary delta and document the
database state through committed reports/tables instead. Reviewers can rebuild
the local index with:

```bash
PYTHONPATH=src .venv/bin/python pipelines/build_v1_dataset.py
```

## Commit Policy

- Do commit code, tests, docs, and small reviewed CSV outputs.
- Do not commit raw PubChem FTP archives.
- Do not commit regenerated bulk-screening parquet outputs unless the repository policy explicitly changes.
- If a future push needs the large-file state reproduced elsewhere, regenerate it from the documented scripts instead of storing it in normal Git history.
